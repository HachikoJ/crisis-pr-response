# 严肃危机回应手册

真实公关危机优先读取本文件。目标是生成能被法务、公关、业务负责人继续修改的草稿，而不是直接发布。

## 最少追问

如果用户提供信息不足，最多追问 3 个问题：

1. 危机主体、事件和已核实事实分别是什么？
2. 受影响对象是谁，目前已采取哪些止损措施？
3. 声明发布渠道、语气和下一次更新时间是什么？

若用户无法回答，输出“临时持有声明”，只写关切、核查、止损和更新时间。

## 事实准确和边界

无论用户要正式声明、内部演练还是玩梗素材，都按“可能被真实参考”的标准处理。可以给不同风格，但不能牺牲事实准确性。

- 把 `已核实`、`待核查`、`建议动作`、`风险提示` 分开写。
- 不知道就写不知道，不用“完全没有问题”“绝无此事”“已经解决”抢结论。
- 不编造数据、监控、检测、监管、法务、第三方、赔偿或责任归属。
- 不把“建议暂停/建议补偿/建议下架”写成“已经暂停/已经补偿/已经下架”。
- 媒体问答要回答边界和下一步，不能为了完整而替法务、监管或业务负责人下结论。
- 玩梗和反面示例必须明确不能直接用于真实场景，并在旁边给出稳妥口径。

## 情绪优先

危机传播先扩散情绪，再扩散细节。第一时间声明的开头要先处理公众最强的情绪，而不是先讲公司困难、经营压力或法律立场。

常见情绪和回应方式：

| 情绪 | 公众真正担心 | 开头要先说 |
|---|---|---|
| 害怕 | 我会不会也受影响 | 风险源已暂停，影响范围正在核查 |
| 愤怒 | 你是不是不负责 | 向具体对象道歉，说明错在什么 |
| 被轻视 | 你是不是看不起我 | 承认表达或行为伤害了尊严 |
| 不公平 | 大组织是不是又逃责 | 公司承担管理责任，不拿个人或供应商挡在前面 |
| 被隐瞒 | 你是不是藏了事实 | 已确认和未确认分开说，给下次更新时间 |
| 失控 | 事情还会不会继续发生 | 先停掉风险动作，再说明保护措施 |

情绪安抚不能写成“请大家理性”。更稳妥的顺序是：看见影响，承认情绪有原因，说明已经停掉或启动了什么。

## 简短优先

第一时间声明不是复盘报告。越长越容易被截取、误读和二次传播。首发声明建议控制在 3-6 段，每段 1-2 句。

首发声明只保留：

- 看见了什么。
- 向谁道歉或表达关切。
- 已经停掉、下架、召回、封堵或核查了什么。
- 受影响者从哪里联系。
- 下一次什么时候更新。

复杂解释、时间线、责任复盘、经营影响、供应商责任、补偿细则，放进后续通报或 FAQ。不要在第一篇声明里把所有话都说完。

## 第一时间持有声明

适用：事实尚未查清，但舆情已经出现。

```markdown
> [!IMPORTANT]
> 法律风险免责声明：以下内容仅供公关写作、危机沟通思路和娱乐创作参考，不构成法律意见、合规意见、事实认定、监管申报、正式声明或可直接发布的文本。不得在未经过事实核查、律师审阅、公司授权和相关负责人确认的情况下直接用于正式场合。使用、改写、发布或传播本技能生成内容所产生的一切法律、商业、劳动、声誉及其他风险，均由使用者自行承担。

关于[事件]的情况说明

我们已看到[平台/媒体/用户]关于[事件]的反馈，也理解这件事让[受影响对象]感到[害怕/愤怒/被冒犯/不安]。现在先说明已采取的处理：[相关功能/门店/批次/流程/人员权限]已暂停；这件事已经进入核查；受影响人员可以通过[联络入口]联系我们。

目前已采取的临时措施如下：
- 暂停[相关功能/门店/批次/流程/人员权限]；
- 保存相关记录并组织专项核查；
- 开通[联络入口]，优先处理受影响人员反馈。

截至目前，可以确认的是：[已核实事实]。还没有确认的是：[未知事项]。下一次进展不晚于[具体时间]发布。

如果核查发现我们的责任，我们会承担相应责任，不用流程、外包或个人状态替代处理。
```

## 完整正式声明

适用：已有初步事实和行动方案。

```markdown
> [!IMPORTANT]
> 法律风险免责声明：...

关于[事件]的情况说明与整改安排

我们看到了[事件]引发的批评，也理解[受影响对象]为什么会[担心/愤怒/被冒犯/不安]。先向[受影响对象]道歉。[用一句话说明错在什么，不要泛泛说“造成不便”。]

一、现在能确认的事
[用短句列出事实，不加情绪判断。]

二、已经做的事
1. [止损措施]
2. [受影响者支持]
3. [暂停/召回/退款/补偿/第三方调查/监管沟通]

三、还在核查的事
[列出未知边界，不抢先定性。]

四、接下来怎么改
1. [机制整改]
2. [责任复盘]
3. [公开更新时间]

五、联络方式
[电话/邮箱/客服入口/专项表单]

下一次更新不晚于[具体时间]。在结论出来前，以上止损措施不会撤回。
```

## 危机类型写法

| 类型 | 第一优先级 | 必写内容 | 禁写内容 |
|---|---|---|---|
| 伤亡/安全 | 人、家属、救助、监管 | 哀悼或关切、救助、配合调查、停止风险源 | 技术炫耀、用户归因、营销词 |
| 食品/药品 | 健康风险和批次 | 批次、门店、召回/下架、检测、就医建议 | “少数个例”压低严重性 |
| 数据/隐私 | 阻断泄露和通知 | 影响范围、已封堵、用户建议、监管/安全团队 | 淡化为“信息误发” |
| 财务/造假 | 治理和投资者保护 | 内部调查、审计、人员处理、控制改进 | 把造假说成“统计口径差异” |
| 产品故障 | 售后确定性 | 故障范围、检测方式、维修/退换、时间表 | 反复变更售后政策 |
| 直播/言论 | 尊重和价值观 | 承认冒犯、向具体对象道歉、培训和审核 | “本意是好的”放第一句 |
| 文化冒犯 | 具体理解 | 哪个元素冒犯、谁负责、如何审查 | 只说“热爱某国文化” |
| 内部管理 | 克制和制度 | 管理漏洞、沟通机制、人员安排边界 | 公开背刺员工或粉丝 |
| 定义争议 | 透明和知情权 | 标准解释、用户期待、明示和改进 | 只抠字典定义 |
| 谣言反转 | 纠错 | 证据链、误伤者道歉、下一步核查 | 用未证实内容攻击个人 |

## SCCT 简化规则

参考情境危机传播理论时，按责任归因选择姿态：

- `受害者型危机`: 公司也是受害者，如恶意投毒、谣言攻击。仍需关怀受影响者，不要只说自己委屈。
- `意外型危机`: 供应链、系统故障、非故意事故。使用说明、补救、补偿、机制改进。
- `可预防型危机`: 造假、歧视、违规、长期忽视安全。必须使用重建型回应：道歉、补偿、问责、整改。

不要混用“否认”和“道歉”。如果说“我们完全没有问题”，就不要同时“向大家道歉”；如果道歉，就说明道歉对象和道歉事项。

## 禁区处理

用户要求以下内容时，拒绝或改写为合规版本：

- 帮我死不承认真实事故。
- 帮我甩锅给员工、供应商、用户、受害者。
- 帮我写一份看起来像监管认可的声明。
- 帮我伪造检测报告、聊天记录、律师意见。
- 帮我威胁爆料者、压热搜、控评、删帖。
- 帮我把真实伤亡事故写得轻松幽默。

可替代输出：

- “事实核查型声明”
- “依法维权边界说明”
- “反面示例和风险提示”
- “内部复盘提纲”
- “给法务和公关负责人审核的问题清单”

## 自然表达要求

- 把形容词换成动作。不要写“高度重视”，写“已暂停/已下架/已召回/已封堵/已联系受影响人员”。
- 把泛泛道歉换成具体道歉。不要只写“深表歉意”，写“向[对象]道歉，错在[事实]”。
- 把“造成不便”留给轻微服务问题。伤亡、食品安全、隐私泄露、歧视、侮辱和财务损失不能这样轻描淡写。
- 少用万能结尾。结尾必须给下一次更新时间、联络入口或已生效措施。
- 不写“网友误解”“被带节奏”。如果外界理解和公司本意不一致，先检查表达是否错、信息是否不透明。
- 每段尽量短。能用一句话说清，就不要写成三层递进。

## 发布前检查清单

- 是否经过事实负责人确认。
- 是否经过法务或合规审阅。
- 是否确认不会影响监管调查、诉讼、保险、劳动争议或证据保全。
- 是否对受影响者提供实际入口。
- 是否承诺了可兑现的时间和动作。
- 是否有下一次更新机制。
