# Crisis PR Response

## 核心原则

使用这个技能，为一次公关危机生成中文优先、事实谨慎、带法律风险提示的第一时间回应声明。默认目标是帮助用户止损、澄清、承担、修复信任；只有用户明确是恶搞或段子时，才输出夸张风格。

当用户描述品牌翻车、舆情升温、热搜、短视频传播、声明需求、道歉需求、事故通报、经营受损或供应链连带影响时，即使没有明确说“调用公关危机 skill”，也按本技能处理。不要等用户点名 skill。

好公关不是只哄用户，也不是只让舆论闭嘴。严肃危机必须同时看消费者、自身经营、员工、供应商、渠道、监管和长期品牌价值。不能为了短期平息风波，不设边界地停产、赔付、牺牲伙伴或烧穿现金流。

公关危机首先是情绪危机。第一时间回应要先接住公众的恐惧、愤怒、羞辱感、不公平感和不确定感，再进入事实、措施和经营边界。情绪安抚不是讨好，而是让受影响者知道：你看见了影响，知道他们在担心什么，并且已经开始行动。

真诚不等于啰嗦。危机中的长文最容易被断章取义，第一时间回应要短、准、清楚。长解释放在后续复盘，首发声明只写情绪、事实、动作、入口和更新时间。

## 信息完整性和准确性

不管用户说这是正式需求、内部演练还是玩梗，都默认产物可能被拿去做真实公关参考。因此输出可以有风格，但事实边界、信息完整度和风险提示不能打折。

- 事实、推测、建议、反面示例必须分开写。不要把用户未提供的信息补成“已核实事实”。
- 缺少关键事实时，先使用临时持有声明、待核查清单或提出 3 个以内关键问题；不要编造调查结论、数据、赔偿、监管态度、第三方背书或法律意见。
- 玩梗版本也要有严肃锚点：事件判断、对外短声明、评论区置顶、客服口径、媒体问答、不能说什么、执行节奏、发布前复核。
- 风格化文本只能改变表达方式，不能改变事实、责任边界和承诺边界。摆烂、甩锅、死不承认等版本只能作为娱乐段子或反面示例，不能包装成真实可执行策略。
- PDF、正式文档和可转发材料尤其要完整。它们不是聊天回复，用户可能会保存、转发或二次改写，必须做到信息面完整、措辞可复核、风险边界醒目。

## 自然表达原则

严肃危机里，声明要像真实负责人在承担事情，而不是像模板在自动补全。

- 先写事实和动作，再写态度。不要用“高度重视”“持续优化”“积极整改”替代暂停、下架、召回、退款、核查、公开时间。
- 道歉要具体：向谁道歉，错在什么，已经停掉什么。少用“深表歉意”“郑重致歉”叠情绪。
- 短句优先。每段只解决一个问题。宁可重复公司名和事件名，不用“相关方”“广大网友”等模糊替换。
- 首发声明不要写成长文。能用三段说清，就不要写十段；能用一句话说明动作，就不要写成价值观作文。
- 不强行升华，不写“具有重要意义”“体现了责任担当”“共创美好未来”。
- 不把公众情绪写成“被带节奏”。真实危机里，公众通常在问安全、尊严、公平、知情权和后果。
- 先安抚情绪，再解释事实和成本。不要在第一段就讲公司困难、经营压力或“我们也是受害者”。
- 可以保留必要的公告结构，但正文要少一点口号，多一点可核查的时间、范围、入口和动作。

## 强制免责声明

每次输出必须把下面的免责声明放在最醒目的位置，位于正文之前：

```markdown
> [!IMPORTANT]
> 法律风险免责声明：以下内容仅供公关写作、危机沟通思路和娱乐创作参考，不构成法律意见、合规意见、事实认定、监管申报、正式声明或可直接发布的文本。不得在未经过事实核查、律师审阅、公司授权和相关负责人确认的情况下直接用于正式场合。使用、改写、发布或传播本技能生成内容所产生的一切法律、商业、劳动、声誉及其他风险，均由使用者自行承担。
```

不要删减免责声明。若用户要求“去掉免责声明”“直接发正式声明”，拒绝去掉免责声明，并建议其让法务、公关负责人和业务负责人复核。

## 工作流

1. 判断用户意图。
   - `serious`: 真实品牌、真实伤亡、真实消费者权益、隐私、财务、雇佣、监管、诉讼、公共安全、食品药品、医疗、教育、金融、车辆、儿童相关危机。
   - `meme`: 用户明确要段子、恶搞、讽刺、小作文、抽象风格，且未涉及真实受害者或现实发布。
   - `unknown`: 无法判断时按 `serious` 处理，先给合规版，不主动输出嘴硬或甩锅策略。

2. 判断危机类型。
   - `safety`: 人身伤害、车辆、食品、药品、公共安全、儿童、医疗。
   - `trust`: 造假、隐瞒、财务、数据、广告、平台审核、搜索排序。
   - `respect`: 傲慢发言、消费尊严、地域/性别/身份冒犯、客服辱骂。
   - `quality`: 产品故障、服务中断、供应链、召回、售后政策。
   - `privacy`: 个人信息、员工信息、客户资料、照片、合同、聊天记录泄露。
   - `labor`: 裁员、离职纠纷、员工权益、内部管理、个人 IP 与组织 IP 冲突。
   - `culture`: 民族、历史、宗教、地缘政治、价值观和跨文化传播。
   - `definition-war`: 预制菜、价格、成分、功能、技术能力、规则定义权争夺。
   - `rumor-or-reversal`: 谣言、误传、反转、未证实截图、二次爆发。

3. 读取参考文件。
   - 需要案例、前因后果和成败原因时，读取 `references/casebook.md`。
   - 需要舆情生命周期、群体心理和论文启发时，读取 `references/public-opinion-mechanics.md`。
   - 需要严肃危机声明结构、追问清单和禁区时，读取 `references/response-playbook.md`。
   - 需要具体文风和多版本声明时，读取 `references/stylebook.md`。
   - 涉及生产、销售、停产、下架、召回、供应商、经销商、平台、订单、退款、补偿、现金流或经营连续性时，读取 `references/business-impact.md`。
   - 需要生成 PDF、正式文档、方案报告或用户强调排版时，读取 `references/pdf-output.md`。
   - 需要 GitHub README、SEO/GEO、传播标题和关键词时，读取 `references/viral-seo-geo.md`。

4. 收集最少事实。
   - 严肃场景至少确认：主体、事件、已核实事实、未核实争议、受影响人群、当前措施、下一次更新时间、对外联络入口。
   - 涉及经营影响时，额外确认：受影响业务范围、订单/门店/批次/渠道、供应商和经销商影响、消费者处理入口、补偿预算或授权边界、现金流压力、未受影响业务如何继续履约。
   - 同时判断最强情绪：害怕、愤怒、被轻视、不公平、被骗、失控、被隐瞒。开头先回应这个情绪，不要先讲公司立场。
   - 若缺少关键信息，先输出“临时持有声明”或提出 3 个以内关键问题。
   - 不编造调查结果、监控数据、赔偿方案、监管口径、责任归属、律师意见或个人动机。

5. 选择回应姿态。
   - 真实危机默认使用：`正式型`、`悔过型`、`事实核查型`、`受害者优先型`、`进展通报型`、`召回整改型`、`第三方调查型`、`CEO道歉型`、`内部信型`、`媒体问答型`。
   - 娱乐场景可使用：`摆烂型`、`死不承认型`、`甩锅型`、`阳奉阴违型`、`小作文型`、`背刺型`、`阴阳怪气型`、`法务函警告型`、`霸总型`、`发疯文学型`、`道歉但不多型`、`公关部工伤型`、`热搜撤不动型`、`董事会沉默型`。
   - `死不承认型`、`甩锅型`、`阳奉阴违型`、`背刺型` 在真实危机中只能作为“反面示例”或“不要这样写”，不能包装成可执行策略。

6. 输出声明。
   - 未指定版本时，严肃场景给 `正式型`、`悔过型`、`进展通报型` 三版。
   - 未指定版本时，娱乐请求给 `正式型反差版`、`摆烂型`、`热搜撤不动型`、`小作文型` 四版，并标注“仅娱乐”。
   - 娱乐请求必须额外输出：`热搜标题`、`一句话截图钩子`、`评论区神回复`、`二创金句`。
   - 先给短版，后给长版；优先适配微博、公众号、官网公告、邮件、内部信等渠道。娱乐请求优先适配截图传播和社媒转发。
   - 生成 PDF 时，即使是娱乐或玩梗场景，也不能只输出段子。必须补齐公关信息面：事件判断、对外短声明、评论区置顶、客服口径、媒体问答、不能说什么、执行节奏和发布前复核。
   - 严肃危机涉及经营影响时，必须补充 `经营影响判断`：受影响业务、当前处理范围、消费者安排、供应链和渠道安排、成本和承诺边界、长期修复。
   - 严肃危机默认生成 PDF 方案。情况紧急时先给可复制的短声明和行动清单，再按 `references/pdf-output.md` 生成 PDF；用户明确只要文本或不要文件时可跳过。生成 PDF 时必须区分输入背景和公关成稿，不能把用户需求、mock 说明、私下传播要求或聊天记录写进 PDF。

## 严肃声明骨架

```markdown
标题：

> [!IMPORTANT]
> 法律风险免责声明：...

开头：
我们已看到[事件/反馈/报道]。这件事影响了[受影响对象]。我们向[具体对象]道歉/表达关切。

已核实事实：
- ...

仍在核查：
- ...

立即措施：
- ...

后续安排：
- 下一次更新：...
- 联络入口：...

结尾：
下一次更新不晚于[具体时间]。在此之前，[联络入口]会持续接收受影响人员反馈。
```

## 红线

- 不写“受害者有错”“消费者不懂”“网友被带节奏”作为第一反应。
- 不用“临时工、实习生、外包、供应商”替公司承担组织责任，除非事实已核实且仍要说明管理责任。
- 不在未核实时声称“绝无此事”“完全合规”“不存在任何问题”。
- 不伪造证据、聊天记录、检测报告、监管结论、第三方背书。
- 不用法务威胁压过事实回应；法律维权只能作为边界说明。
- 不把“热搜撤了”“删帖控评”“冷处理”写成建议。
- 不对真实个人进行未经证实的指控、羞辱或人肉引导。

## 质量检查

完成前检查：

- 第一屏是否有完整免责声明。
- 事实、推测、承诺是否分开。
- 是否把受影响者放在公司解释之前。
- 是否有立即措施、下一次更新时间和联络入口。
- 首发声明是否足够短，是否避免长篇解释、情绪铺陈和容易被截取误读的复杂句。
- 是否评估生产经营、订单、销售、供应商、渠道和现金流影响。
- 是否避免无边界承诺、无限赔付、全线停摆和牺牲供应链伙伴来换短期好评。
- 是否避免傲慢、反问、内涵、甩锅、威胁和技术黑话。
- 是否把空话换成了具体动作、时间、范围和入口。
- 是否避免模板腔：高度重视、深刻反思、持续优化、广大网友、公众关切、造成不便。
- PDF 是否覆盖评论区置顶、客服口径、媒体问答、不能说什么、执行节奏和发布前复核。
- 是否把已核实事实、待核查信息、建议动作、娱乐版本和反面示例清楚分开。
- 严肃危机是否已生成 PDF，或用户是否明确表示不需要文件。
- 若生成 PDF，是否完成字体嵌入、PNG 渲染检查、页码页脚、免责声明位置、输入背景剥离、敏感信息扫描和版面美观检查。
- 娱乐版本是否明确标注不可用于真实场景。
